## Cause 2 — View column/type mismatch (data present but `SELECT *` fails) — NOT PURSUED

> Disposition: owner decision — not remediating. Root cause confirmed: the `type`
> partition column is a per-table constant, materialized on some tables but not others,
> so the view/rowType declares one more column than the parquet holds. Blast radius of a
> framework-level fix is too large. Tables stay populated-but-unreadable via `SELECT *`.

`REACH=OK` with real row counts, but the row fetch fails:
`Binder Error: Contents of view were altered: types don't match` /
`unnamed_subquery has N columns available but N+1 columns specified`.
The DuckDB view wrapping the Iceberg table has a column-list/type mismatch vs the physical
parquet, so these populated tables are **unreadable via a normal client `SELECT *`**.

- **econ_reference:** `fred_series`, `jolts_dataelements`, `bls_geographies`, `nipa_tables`, `regional_linecodes`, `naics_sectors`, `jolts_industries`
- **health:** `fda_ndc_products` (136,990 rows), `medicaid_drug_utilization` (15.9M rows)
- **transport:** `safety_complaints` (20M rows)
